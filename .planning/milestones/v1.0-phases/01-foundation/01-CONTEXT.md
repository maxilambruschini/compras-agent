# Phase 1: Foundation - Context

**Gathered:** 2026-05-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 1 delivers the complete data contract that all subsequent phases depend on:
- Docker Compose scaffold with FastAPI, Postgres, and a Vite+React frontend container
- Postgres schema with Alembic migrations (auto-run on startup)
- SQLAlchemy ORM models for DB schema definition and queries
- Pydantic extraction models (`ExtractedInvoice` + `LineItem`) as the GPT-4o output contract
- Environment variable loading with startup validation (app refuses to start if required vars are missing)
- Allowlist table for sender phone numbers (INF-01)

**This phase does NOT include:** any AI extraction logic, WhatsApp webhook handling, or real UI screens — those are Phases 2, 3, and 4 respectively.

</domain>

<decisions>
## Implementation Decisions

### Project Layout
- **D-01:** Repo organized as `backend/` + `frontend/` sibling directories at the repo root.
- **D-02:** Python package lives at `backend/app/` — conventional FastAPI layout with subdirs: `app/models/`, `app/routers/`, `app/services/`, `app/db/`.
- **D-03:** Alembic manages its own directory structure — initialize with `alembic init` from `backend/`, which places `alembic/` and `alembic.ini` under `backend/`.

### Database Migrations
- **D-04:** Alembic is the migration tool. SQLAlchemy ORM models drive schema definition; migrations are autogenerated via `alembic revision --autogenerate`.
- **D-05:** Migrations run automatically on FastAPI container startup (`alembic upgrade head` runs before Uvicorn starts) — `docker compose up` always leaves the schema current.
- **D-06:** SQLAlchemy ORM (`AsyncSession`) is used for all DB queries in FastAPI routes, not just schema definition.

### Pydantic Extraction Models
- **D-07:** `ExtractedInvoice` is nested: document-level header fields + `line_items: List[LineItem]`. Mirrors the REQUIREMENTS.md EXT-01/EXT-02 (line items) vs EXT-03/EXT-04 (document fields) split.
- **D-08:** Argentine invoice types use a Python `str` Enum: `TipoComprobante` with values `FACTURA_A`, `FACTURA_B`, `FACTURA_C`, `REMITO`, `LISTA_INFORMAL`, `UNKNOWN`. String enum so Postgres stores readable labels.
- **D-09:** Pydantic extraction models live in `backend/app/models/extraction.py`. SQLAlchemy ORM models live in `backend/app/db/models.py`. Two separate files, two separate concerns.

### Frontend (Stub)
- **D-10:** Frontend is a real Vite + React scaffold (not a static placeholder) — established in Phase 1, built out in Phase 4.
- **D-11:** Package manager is **pnpm** — not npm or yarn. All `frontend/` package operations use `pnpm`.
- **D-12:** Frontend container runs `pnpm dev` (Vite dev server, hot-reload, port 5173). Source is volume-mounted so changes reflect without container rebuild.

### Claude's Discretion
- Internal Python package subdir structure within `backend/app/` — Claude picks the conventional FastAPI layout (models/, routers/, services/, db/).

</decisions>

<specifics>
## Specific Ideas

- User explicitly requested **pnpm** over npm — "it's much better." This applies to all frontend operations: install, dev, build, and in the Dockerfile.
- `TipoComprobante` enum must include `UNKNOWN` as a fallback — GPT-4o may encounter document types not in the enum.
- All Pydantic extraction fields must be `Optional[T]` with `default=None` — EXT-06 requires null over hallucination. This is a hard constraint on the model definition.
