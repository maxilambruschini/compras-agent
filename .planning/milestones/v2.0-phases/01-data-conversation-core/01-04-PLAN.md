<objective>
Build the ConversationOrchestrator — the deterministic match-based state machine that owns every transition and the DB write gate. It wires together SlotExtractionService (Plan 02), GastoService (Plan 03), and the Conversation/Gasto models (Plan 01) behind a per-sender row lock, DB-backed idempotency, and timeout reset.

Purpose: This is the heart of the conversation engine and the plan that satisfies the bulk of the ROADMAP success criteria. It depends on all three earlier plans (01-01 Wave 1, 01-02 Wave 2, 01-03 Wave 3), so it runs alone in Wave 4 — it is the last link in the strictly sequential chain 01-01 → 01-02 → 01-03 → 01-04, matching this plan's frontmatter `wave: 4`.

Output: ConversationOrchestrator + a comprehensive test file covering CONV-02/03/04/06 and GASTO-02/04/06 plus the confirm gate (GASTO-05). TDD plan — write the behavior tests first.

**Lock-strength contract (review concern 2 — verified, not assumed):** The required Postgres lock is `SELECT ... FOR NO KEY UPDATE` (success criterion 3). In SQLAlchemy 2.0.49 (this project's pinned version), `select(...).with_for_update(key_share=True)` compiles to exactly `FOR NO KEY UPDATE` under the postgresql dialect — this was empirically verified during replanning. (`key_share=False` → `FOR UPDATE`; `read=True, key_share=True` → `FOR KEY SHARE`; `read=True` → `FOR SHARE`.) The plan therefore KEEPS `with_for_update(key_share=True)` and pins the contract with a compiled-SQL assertion in Task 0 so a SQLAlchemy upgrade cannot silently change the emitted lock mode. Do NOT switch to `key_share=False` — that emits the stronger `FOR UPDATE`, not the required `FOR NO KEY UPDATE`.

**Missing-row concurrency contract (review cycle-2 HIGH — get-or-create before lock):** `SELECT ... FOR NO KEY UPDATE` can only lock a row that already exists. For a brand-new sender, no `conversations` row exists yet, so two concurrent FIRST messages from that sender are NOT serialized by the lock — they race on the `sender_phone` primary-key INSERT (one wins, the other raises `IntegrityError`). The plan therefore makes row materialization race-safe with a Postgres-native upsert BEFORE the lock: `INSERT INTO conversations (sender_phone, state, ...) VALUES (...) ON CONFLICT (sender_phone) DO NOTHING` (via `sqlalchemy.dialects.postgresql.insert(...).on_conflict_do_nothing(index_elements=["sender_phone"])`), then `SELECT ... FOR NO KEY UPDATE` to lock the now-guaranteed-to-exist row. The losing INSERT no-ops instead of erroring, and both transactions converge on the single row under the lock. The orchestrator's entry sequence becomes: **ensure-row-exists (race-safe ON CONFLICT DO NOTHING) → lock row (FOR NO KEY UPDATE) → idempotency check → timeout → dispatch → commit → reply-outside-txn.** `ON CONFLICT` is also valid SQLite syntax, so the get-or-create path is unit-testable on SQLite (a second insert no-ops without error). However, TRUE cross-session serialization of two concurrent first-message inserts is a Postgres-only behavior — SQLite's in-process test engine cannot reproduce a genuine cross-connection insert race, so that proof is a separately-marked Postgres integration test deferred to verify-phase (see test artifact named in the threat model / VALIDATION deferred row).
</objective>
