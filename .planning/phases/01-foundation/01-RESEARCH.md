## Summary

Phase 1 establishes the entire data contract and execution scaffold for the project. The three most technically nuanced areas are: (1) the Alembic `env.py` async configuration, which requires `asyncio.run()` wrapping a `run_sync` call — the single most common footgun in this stack; (2) running `alembic upgrade head` before Uvicorn inside the Docker entrypoint command rather than inside FastAPI's lifespan — this avoids the indefinite-hang issue documented in alembic/alembic#1483; and (3) the Docker Compose dependency chain requiring Postgres `service_healthy` (not just `service_started`) before the backend boots.

The Walking Skeleton goal — prove all four tiers wire together — is achievable by implementing one real DB write path: a `GET /health` endpoint that reads from the `sender_allowlist` table, returning count=0. This proves FastAPI → SQLAlchemy async session → Postgres round-trip and Alembic migrations ran without touching the AI or WhatsApp subsystems.

**Primary recommendation:** Use the Docker entrypoint command pattern (`alembic upgrade head && uvicorn ...`) rather than FastAPI lifespan for migrations. This is the canonical approach for containerized deployments and avoids async-inside-sync footguns.

### Pattern 1: Alembic async env.py

**What:** Alembic's default `env.py` uses synchronous `engine.connect()`. For async drivers (asyncpg), this must be replaced with `async_engine_from_config` + `asyncio.run()`.

**When to use:** Always when the application uses `postgresql+asyncpg://` URLs.

**The footgun:** Forgetting to call `asyncio.run()` causes `run_migrations_online()` to silently do nothing or hang. The synchronous migration code path CANNOT use `await` directly — it must be wrapped.

```python
# backend/alembic/env.py
# Source: https://alembic.sqlalchemy.org/en/latest/cookbook.html (async section)
import asyncio
from logging.config import fileConfig

from sqlalchemy import pool
from sqlalchemy.engine import Connection
from sqlalchemy.ext.asyncio import async_engine_from_config

from alembic import context

# Import ALL ORM models so autogenerate can see them
from app.db.models import Base  # noqa: F401

config = context.config
if config.config_file_name is not None:
    fileConfig(config.config_file_name)

target_metadata = Base.metadata


def do_run_migrations(connection: Connection) -> None:
    context.configure(connection=connection, target_metadata=target_metadata)
    with context.begin_transaction():
        context.run_migrations()


async def run_async_migrations() -> None:
    connectable = async_engine_from_config(
        config.get_section(config.config_ini_section, {}),
        prefix="sqlalchemy.",
        poolclass=pool.NullPool,  # NullPool required for migration scripts
    )
    async with connectable.connect() as connection:
        await connection.run_sync(do_run_migrations)
    await connectable.dispose()


def run_migrations_online() -> None:
    asyncio.run(run_async_migrations())


def run_migrations_offline() -> None:
    url = config.get_main_option("sqlalchemy.url")
    context.configure(
        url=url,
        target_metadata=target_metadata,
        literal_binds=True,
        dialect_opts={"paramstyle": "named"},
    )
    with context.begin_transaction():
        context.run_migrations()


if context.is_offline_mode():
    run_migrations_offline()
else:
    run_migrations_online()
```

**Critical detail:** `pool.NullPool` is mandatory in migration scripts — connection pool interference causes hangs when using async engines in subprocess/script context. [CITED: https://alembic.sqlalchemy.org/en/latest/cookbook.html]

### Pattern 2: Docker Compose Entrypoint (migrations before app)

**What:** Run `alembic upgrade head` in the same command as `uvicorn` using `bash -c`.

**When to use:** Always for containerized deployment. Avoids the programmatic `command.upgrade()` indefinite-hang issue.

```yaml
# docker-compose.yml
# Source: [CITED: berkkaraal.com/blog] + [VERIFIED: Docker Compose healthcheck docs]
services:
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: ${POSTGRES_USER}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: ${POSTGRES_DB}
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER} -d ${POSTGRES_DB}"]
      interval: 5s
      timeout: 5s
      retries: 5
    ports:
      - "5432:5432"

  backend:
    build: ./backend
    command: bash -c "alembic upgrade head && uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload"
    ports:
      - "8000:8000"
    volumes:
      - ./backend:/app    # hot-reload in dev; remove in production
    depends_on:
      postgres:
        condition: service_healthy   # CRITICAL: wait for pg_isready, not just container started
    env_file: .env
    environment:
      - DATABASE_URL=postgresql+asyncpg://${POSTGRES_USER}:${POSTGRES_PASSWORD}@postgres:5432/${POSTGRES_DB}

  frontend:
    build: ./frontend
    command: pnpm dev --host 0.0.0.0
    ports:
      - "5173:5173"
    volumes:
      - ./frontend/src:/app/src       # hot-reload source
      - ./frontend/public:/app/public
      - /app/node_modules             # prevent host node_modules from overriding container's

volumes:
  postgres_data:
```

**Critical detail:** `condition: service_healthy` requires the `healthcheck` block on the postgres service. Without it, `depends_on` only waits for container start, not Postgres readiness — backend fails with "connection refused" on first startup. [VERIFIED: Docker Compose docs]

### Pattern 3: SQLAlchemy Async Engine + Session Dependency

**What:** Create engine + session factory once at module level; inject sessions per-request via FastAPI Depends.

```python
# backend/app/db/engine.py
# Source: https://docs.sqlalchemy.org/en/20/orm/extensions/asyncio.html
from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker, AsyncSession
from app.config import get_settings

settings = get_settings()

engine = create_async_engine(
    settings.database_url,
    echo=settings.debug,         # logs SQL in development
    pool_pre_ping=True,          # detect stale connections
)

AsyncSessionLocal = async_sessionmaker(
    engine,
    expire_on_commit=False,      # prevent lazy-load errors after commit
    class_=AsyncSession,
)
```

```python
# backend/app/db/session.py
from collections.abc import AsyncGenerator
from sqlalchemy.ext.asyncio import AsyncSession
from app.db.engine import AsyncSessionLocal


async def get_db() -> AsyncGenerator[AsyncSession, None]:
    async with AsyncSessionLocal() as session:
        try:
            yield session
        finally:
            await session.close()
```

```python
# Usage in router
from fastapi import Depends
from sqlalchemy.ext.asyncio import AsyncSession
from app.db.session import get_db

@router.get("/health")
async def health(db: AsyncSession = Depends(get_db)):
    result = await db.execute(select(func.count()).select_from(SenderAllowlist))
    count = result.scalar()
    return {"status": "ok", "allowlist_count": count}
```

**Critical detail:** `expire_on_commit=False` prevents `MissingGreenlet` errors when accessing ORM attributes after `commit()` inside an async context. [CITED: https://docs.sqlalchemy.org/en/20/orm/extensions/asyncio.html]

### Pattern 4: pydantic-settings for Env Validation

**What:** `BaseSettings` fields with no default value are REQUIRED — if missing from environment, Pydantic raises `ValidationError` at import time, refusing to start.

```python
# backend/app/config.py
# Source: https://fastapi.tiangolo.com/advanced/settings
from functools import lru_cache
from pydantic import PostgresDsn
from pydantic_settings import BaseSettings, SettingsConfigDict


class Settings(BaseSettings):
    model_config = SettingsConfigDict(
        env_file=".env",
        env_file_encoding="utf-8",
        case_sensitive=False,
    )

    # Required — no default — app refuses to start if missing (INF-03)
    database_url: str           # postgresql+asyncpg://user:pass@host:5432/db
    openai_api_key: str
    whatsapp_token: str
    whatsapp_phone_number_id: str
    whatsapp_verify_token: str

    # Optional with defaults
    debug: bool = False
    log_level: str = "INFO"
    confidence_threshold: float = 0.85


@lru_cache
def get_settings() -> Settings:
    return Settings()
```

**Critical detail:** Fields typed as `str` with no `= "default"` are required. If `OPENAI_API_KEY` is absent from environment AND `.env`, `Settings()` construction raises `ValidationError: openai_api_key field required`. This satisfies INF-03 (fail-fast on missing secrets). [CITED: https://fastapi.tiangolo.com/advanced/settings]

### Pattern 5: SQLAlchemy ORM Models for Invoice Schema

```python
# backend/app/db/models.py
# Source: https://docs.sqlalchemy.org/en/20/orm/declarative_styles.html
from __future__ import annotations
import uuid
from datetime import date, datetime
from decimal import Decimal
from typing import List, Optional

from sqlalchemy import (
    ForeignKey, String, Text, Numeric, Date, DateTime,
    Boolean, Integer, func, Index
)
from sqlalchemy.dialects.postgresql import UUID
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column, relationship


class Base(DeclarativeBase):
    pass


class Invoice(Base):
    __tablename__ = "invoices"

    id: Mapped[uuid.UUID] = mapped_column(
        UUID(as_uuid=True), primary_key=True, default=uuid.uuid4
    )
    # Document header (nullable — extraction may be partial)
    tipo_comprobante: Mapped[Optional[str]] = mapped_column(String(50))
    numero_documento: Mapped[Optional[str]] = mapped_column(String(100))
    proveedor: Mapped[Optional[str]] = mapped_column(String(255))
    fecha: Mapped[Optional[date]] = mapped_column(Date)
    cuit_proveedor: Mapped[Optional[str]] = mapped_column(String(13))
    cae: Mapped[Optional[str]] = mapped_column(String(20))
    fecha_vencimiento_cae: Mapped[Optional[date]] = mapped_column(Date)
    # Processing metadata
    confidence_score: Mapped[Optional[float]] = mapped_column(Numeric(4, 3))
    status: Mapped[str] = mapped_column(String(30), default="pending_review")
    whatsapp_message_id: Mapped[Optional[str]] = mapped_column(String(100))
    sender_phone: Mapped[Optional[str]] = mapped_column(String(30))
    image_path: Mapped[Optional[str]] = mapped_column(Text)
    raw_extraction: Mapped[Optional[str]] = mapped_column(Text)  # JSON dump
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), server_default=func.now()
    )
    updated_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), server_default=func.now(), onupdate=func.now()
    )
    # Relationships
    line_items: Mapped[List["InvoiceLineItem"]] = relationship(
        back_populates="invoice", cascade="all, delete-orphan"
    )

    __table_args__ = (
        Index("ix_invoices_numero_documento_proveedor", "numero_documento", "proveedor"),
        Index("ix_invoices_status", "status"),
        Index("ix_invoices_created_at", "created_at"),
    )


class InvoiceLineItem(Base):
    __tablename__ = "invoice_line_items"

    id: Mapped[int] = mapped_column(Integer, primary_key=True, autoincrement=True)
    invoice_id: Mapped[uuid.UUID] = mapped_column(
        UUID(as_uuid=True), ForeignKey("invoices.id", ondelete="CASCADE")
    )
    descripcion: Mapped[Optional[str]] = mapped_column(Text)
    codigo_sku: Mapped[Optional[str]] = mapped_column(String(100))
    bultos: Mapped[Optional[Decimal]] = mapped_column(Numeric(12, 4))
    unidades_por_bulto: Mapped[Optional[Decimal]] = mapped_column(Numeric(12, 4))
    precio_unitario_sin_iva: Mapped[Optional[Decimal]] = mapped_column(Numeric(14, 4))
    descuento_pct: Mapped[Optional[Decimal]] = mapped_column(Numeric(6, 4))
    iva_rate: Mapped[Optional[Decimal]] = mapped_column(Numeric(6, 4))
    percepciones_iibb: Mapped[Optional[Decimal]] = mapped_column(Numeric(14, 4))

    invoice: Mapped["Invoice"] = relationship(back_populates="line_items")


class SenderAllowlist(Base):
    __tablename__ = "sender_allowlist"

    id: Mapped[int] = mapped_column(Integer, primary_key=True, autoincrement=True)
    phone_number: Mapped[str] = mapped_column(String(30), unique=True, nullable=False)
    display_name: Mapped[Optional[str]] = mapped_column(String(100))
    is_active: Mapped[bool] = mapped_column(Boolean, default=True)
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), server_default=func.now()
    )
```

### Pattern 6: Pydantic Extraction Models (ExtractedInvoice)

```python
# backend/app/models/extraction.py
# Source: https://pydantic.dev/docs/validation — verified Optional pattern
from __future__ import annotations
from decimal import Decimal
from enum import Enum
from typing import List, Optional
from pydantic import BaseModel, ConfigDict


class TipoComprobante(str, Enum):
    """Argentine invoice document types. str Enum stores readable labels in Postgres."""
    FACTURA_A = "FACTURA_A"
    FACTURA_B = "FACTURA_B"
    FACTURA_C = "FACTURA_C"
    REMITO = "REMITO"
    LISTA_INFORMAL = "LISTA_INFORMAL"
    UNKNOWN = "UNKNOWN"    # Required fallback — GPT-4o may see unknown types


class LineItem(BaseModel):
    """Extracted line item — all fields Optional per EXT-06 (null > hallucination)."""
    model_config = ConfigDict(use_enum_values=True)

    descripcion: Optional[str] = None
    codigo_sku: Optional[str] = None
    bultos: Optional[Decimal] = None
    unidades_por_bulto: Optional[Decimal] = None
    precio_unitario_sin_iva: Optional[Decimal] = None
    descuento_pct: Optional[Decimal] = None
    iva_rate: Optional[Decimal] = None
    percepciones_iibb: Optional[Decimal] = None


class ExtractedInvoice(BaseModel):
    """Top-level extraction output contract for GPT-4o response parsing.

    All fields are Optional per EXT-06. The UNKNOWN fallback on tipo_comprobante
    ensures the model never fails on unrecognized document types (EXT-05).
    """
    model_config = ConfigDict(use_enum_values=True)

    tipo_comprobante: Optional[TipoComprobante] = None
    numero_documento: Optional[str] = None
    proveedor: Optional[str] = None
    fecha: Optional[str] = None          # ISO 8601 string; parse to date in service layer
    cuit_proveedor: Optional[str] = None
    cae: Optional[str] = None
    fecha_vencimiento_cae: Optional[str] = None
    line_items: List[LineItem] = []
    # confidence_score is computed in Phase 2 ExtractionService, not by GPT-4o
```

**Critical detail:** `TipoComprobante` must inherit from both `str` and `Enum` to be compatible with `client.chat.completions.parse()` JSON Schema generation in Phase 2. Pure `Enum` would fail Pydantic's JSON serialization for OpenAI Structured Outputs. [CITED: CLAUDE.md §Key Architectural Decisions]

### Pattern 7: FastAPI App + Lifespan

```python
# backend/app/main.py
from contextlib import asynccontextmanager
from fastapi import FastAPI
from app.config import get_settings
import structlog

log = structlog.get_logger()


@asynccontextmanager
async def lifespan(app: FastAPI):
    # Startup: validate config (raises on missing env vars)
    settings = get_settings()
    log.info("app.starting", env=settings.log_level)
    # NOTE: alembic upgrade head runs in Docker entrypoint BEFORE this lifespan
    # Do NOT re-run migrations here — leads to alembic#1483 hang
    yield
    # Shutdown: dispose async engine
    from app.db.engine import engine
    await engine.dispose()
    log.info("app.stopped")


def create_app() -> FastAPI:
    settings = get_settings()
    app = FastAPI(
        title="Compras Agent API",
        lifespan=lifespan,
        debug=settings.debug,
    )
    from app.routers.health import router as health_router
    app.include_router(health_router)
    return app


app = create_app()
```

### Anti-Patterns to Avoid

- **Running `command.upgrade(alembic_cfg, "head")` inside FastAPI lifespan:** This can hang indefinitely when called from within an async context. Use the shell entrypoint pattern instead. [CITED: https://github.com/sqlalchemy/alembic/discussions/1483]
- **`depends_on: postgres` without `condition: service_healthy`:** Docker only waits for the container to START, not for Postgres to be ready to accept connections. Backend will crash on first startup with "connection refused". Always add a healthcheck + `condition: service_healthy`.
- **Forgetting `pool.NullPool` in Alembic `env.py`:** Without `NullPool`, async connection pools from the application engine bleed into migration scripts, causing unexpected blocking behavior.
- **Omitting `expire_on_commit=False` in `async_sessionmaker`:** SQLAlchemy's default `expire_on_commit=True` triggers lazy loads after commit. In async context, this raises `MissingGreenlet` because there's no implicit IO allowed outside `await`.
- **Using `Optional[str]` without `= None` on Pydantic extraction models:** In Pydantic v2, `Optional[str]` does NOT imply `default=None`. You MUST write `Optional[str] = None` explicitly for the field to be truly optional.
- **Not importing ORM models in `alembic/env.py`:** Alembic's autogenerate scans `Base.metadata`. If models are never imported, autogenerate sees an empty metadata and generates empty migrations — tables are never created.
- **Frontend `node_modules` volume conflict:** Mounting `./frontend:/app` in Docker without also mounting `/app/node_modules` as an anonymous volume will overwrite the container's installed packages with the (possibly absent or platform-mismatched) host directory.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Async DB sessions per request | Manual session open/close in each handler | `AsyncSessionLocal` + `Depends(get_db)` | FastAPI's DI handles cleanup; manual approach leaks sessions on exceptions |
| Required env var validation | `if not os.getenv("KEY"): sys.exit(1)` | `pydantic-settings` BaseSettings required field | Pydantic gives typed values + good error messages for free |
| DB schema management | `Base.metadata.create_all()` at startup | Alembic migrations | `create_all` cannot handle schema changes; Alembic tracks versions and generates ALTER statements |
| Postgres readiness wait in Docker | `sleep 5` in entrypoint | `pg_isready` healthcheck + `depends_on: condition: service_healthy` | `sleep` is unreliable; healthcheck retries until truly ready |
| Enum string serialization | `enum.value` calls everywhere | `str, Enum` base + `use_enum_values=True` in ConfigDict | Pydantic handles serialization automatically; OpenAI Structured Outputs works natively |

**Key insight:** The async ORM + migrations stack has many well-documented configuration pitfalls. The "obvious" patterns from SQLAlchemy sync docs all silently break under async — use the exact patterns in this research.

## Common Pitfalls

### Pitfall 1: Alembic autogenerate sees no tables
**What goes wrong:** `alembic revision --autogenerate -m "init"` produces a migration with `pass` — no table creation.
**Why it happens:** ORM models (`Invoice`, etc.) were never imported in `alembic/env.py`. Alembic's autogenerate only sees `Base.metadata` — if models weren't imported, metadata is empty.
**How to avoid:** Add `from app.db.models import Base  # noqa: F401` (and any model files) to `env.py` BEFORE `target_metadata = Base.metadata`.
**Warning signs:** Generated migration file has `def upgrade(): pass` and `def downgrade(): pass`.

### Pitfall 2: Postgres not ready when backend starts
**What goes wrong:** Backend container exits with `asyncpg.exceptions.ConnectionRefusedError` or `FATAL: the database system is starting up`.
**Why it happens:** `depends_on: postgres` only waits for the container to start (Docker daemon level), not for Postgres to be ready to accept connections.
**How to avoid:** Add `healthcheck` to the postgres service using `pg_isready`, then set `condition: service_healthy` on the backend's `depends_on`.
**Warning signs:** Works on second `docker compose up` (Postgres was already warm) but fails on first run.

### Pitfall 3: `MissingGreenlet` error after DB write
**What goes wrong:** `sqlalchemy.exc.MissingGreenlet: greenlet_spawn has not been called` when accessing ORM attributes after `await session.commit()`.
**Why it happens:** `expire_on_commit=True` (default) marks all objects as expired after commit. Accessing an expired attribute in async context triggers a lazy load — but there's no async IO context.
**How to avoid:** Set `expire_on_commit=False` in `async_sessionmaker(engine, expire_on_commit=False)`. For objects you need post-commit, refresh explicitly with `await session.refresh(obj)`.
**Warning signs:** Error appears only after `commit()`, not during query.

### Pitfall 4: Pydantic v2 `Optional` without default
**What goes wrong:** `ExtractedInvoice(tipo_comprobante=None)` raises `ValidationError: field required`.
**Why it happens:** In Pydantic v2, `Optional[str]` is a type annotation meaning `str | None`. It does NOT set a default. A field with no default is still required.
**How to avoid:** Always write `field_name: Optional[Type] = None` for every nullable extraction field.
**Warning signs:** Works in Pydantic v1 (which treated `Optional` as implying `None` default), breaks after v2 migration.

### Pitfall 5: Frontend `node_modules` shadow mount
**What goes wrong:** Vite fails to start in container with "Cannot find module 'vite'" or similar.
**Why it happens:** Docker Compose mounts `./frontend:/app`, overwriting `/app/node_modules` (installed by `pnpm install` in Dockerfile) with the host directory that may not have `node_modules` or has Mac-architecture binaries.
**How to avoid:** Add an anonymous volume for `node_modules`:
```yaml
volumes:
  - ./frontend/src:/app/src
  - ./frontend/public:/app/public
  - /app/node_modules   # anonymous volume preserves container-installed packages
```
**Warning signs:** `pnpm dev` works on host but fails inside container.

### Pitfall 6: alembic.ini `sqlalchemy.url` hardcoded
**What goes wrong:** Migration command ignores `DATABASE_URL` env var and uses the hardcoded URL in `alembic.ini`.
**Why it happens:** Alembic reads `sqlalchemy.url` from `alembic.ini` by default. In `env.py` you must override this with the environment variable.
**How to avoid:** In `env.py`, add before `target_metadata`:
```python
import os
config.set_main_option("sqlalchemy.url", os.environ["DATABASE_URL"])
```
Or use the Settings object if importable from the backend package.
**Warning signs:** Migrations connect to wrong host or fail with auth error despite correct env var.

## Code Examples

### Walking Skeleton Health Endpoint
```python
# backend/app/routers/health.py
from fastapi import APIRouter, Depends
from sqlalchemy import select, func
from sqlalchemy.ext.asyncio import AsyncSession
from app.db.session import get_db
from app.db.models import SenderAllowlist

router = APIRouter()


@router.get("/health")
async def health(db: AsyncSession = Depends(get_db)):
    """Walking skeleton: proves FastAPI -> AsyncSession -> Postgres round-trip works."""
    result = await db.execute(
        select(func.count()).select_from(SenderAllowlist)
    )
    count = result.scalar_one()
    return {"status": "ok", "allowlist_count": count, "db": "connected"}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `Session` (sync) + `create_engine` | `AsyncSession` + `create_async_engine` + asyncpg | SQLAlchemy 1.4+ (stable in 2.0) | All FastAPI route handlers must be `async def`; blocking calls forbidden |
| `declarative_base()` function | `class Base(DeclarativeBase): pass` | SQLAlchemy 2.0 | Better type inference; `Mapped[]` annotations fully supported |
| `Column(String)` | `mapped_column(String)` + `Mapped[Optional[str]]` | SQLAlchemy 2.0 | Full type annotation support; IDEs infer column nullability |
| `sessionmaker` | `async_sessionmaker` | SQLAlchemy 2.0 | Async-specific factory; do not mix with sync sessionmaker |
| FastAPI `@app.on_event("startup")` | FastAPI `lifespan` context manager | FastAPI 0.93+ | `on_event` is deprecated; lifespan is the current standard |
| `BaseSettings` in pydantic | `BaseSettings` in pydantic-settings (separate package) | Pydantic v2 | Settings were extracted to `pydantic-settings`; must install separately |

**Deprecated/outdated:**
- `declarative_base()` from `sqlalchemy.ext.declarative`: Replaced by `class Base(DeclarativeBase)`. Still works but IDEs won't infer types.
- `@app.on_event("startup")` / `@app.on_event("shutdown")`: Deprecated since FastAPI 0.93. Use `lifespan=` parameter instead.
- `from pydantic import BaseSettings`: Moved to `pydantic-settings` package in Pydantic v2. Import from `pydantic_settings` now.
