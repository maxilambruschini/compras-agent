### Plan 01: Data Contract (Pydantic + SQLAlchemy + Alembic + Tests)

**Summary**

A carefully engineered plan that correctly sequences the data contract setup and proves it with automated tests. The research-driven patterns (async env.py, NullPool, expire_on_commit=False) are all present. The main structural flaw is a dependency inversion in Task 1's verify step: `conftest.py` is written in Task 1 but imports `from app.db.models import Base`, which doesn't exist until Task 2. pytest loads conftest at session start, so the Task 1 verify command will fail even though none of the Task 1 tests use DB fixtures.

**Strengths**

- Exact version pinning from verified research — no drift risk
- All RESEARCH.md pitfalls explicitly addressed (NullPool, asyncio.run, Base import in env.py, URL override)
- INF-03 tested at unit level (test_config.py) and deferred for E2E validation to Plan 02
- Wave 0 test gap list directly mapped to files — no ambiguity about what to build
- Alembic URL override pattern (`os.environ.get("DATABASE_URL")`) correctly avoids hardcoding in alembic.ini
- `expire_on_commit=False` called out in both engine.py and conftest fixtures — common omission

**Concerns**

- **HIGH: conftest.py/Task 1 verify ordering bug.** Task 1 instructs creating `conftest.py` with `from app.db.models import Base`, then verifies with `pytest tests/test_config.py tests/test_extraction_models.py -x -q`. pytest loads `conftest.py` at collection time regardless of which test files are selected. Since `app/db/models.py` doesn't exist until Task 2, the entire test session will fail with `ModuleNotFoundError`. Fix: either (a) create a minimal `app/db/__init__.py` and `app/db/models.py` stub in Task 1 with just `Base` and `SenderAllowlist` so conftest can import, or (b) split conftest into two files: a `conftest_base.py` with only env/config fixtures (Task 1) and the DB engine fixtures (Task 2).

- **HIGH: `engine.py` module-level `get_settings()` breaks test imports.** The pattern `settings = get_settings()` at module level in `engine.py` means any file that imports from `app.db.engine` at module level will call `Settings()` at import time — before pytest fixtures have patched env vars. The `test_health.py` instruction says "lazily import app inside the test function body" but this is fragile and easy to get wrong. Fix: call `get_settings()` inside a function or use a lazy-init pattern with `functools.cached_property`, or make the engine a module-level lazy singleton (initialized on first access).

- **MEDIUM: PostgreSQL-specific UUID type incompatible with aiosqlite tests.** Models use `from sqlalchemy.dialects.postgresql import UUID` with `as_uuid=True`. SQLAlchemy does have a fallback rendering for SQLite (stores as CHAR(32)), but `UUID(as_uuid=True)` does the Python↔DB UUID conversion at the DBAPI level — asyncpg handles this, aiosqlite does not. At test time, inserting a `uuid.UUID` object into an aiosqlite column may raise `InterfaceError: Error binding parameter`. Fix: use `sqlalchemy.Uuid` (the dialect-agnostic type introduced in SQLAlchemy 2.0) instead of `sqlalchemy.dialects.postgresql.UUID`. This is a drop-in replacement that works with both asyncpg and aiosqlite.

- **MEDIUM: Alembic autogenerate via SQLite produces SQLite DDL.** Generating the initial migration with `DATABASE_URL=sqlite+aiosqlite:///./_autogen.db alembic revision --autogenerate` will produce SQLite-specific column types (CHAR for UUIDs, REAL for Numeric, etc.) rather than Postgres DDL. The migration will work in containers only if Alembic's `render_as_batch` is not set — but the autogenerated `op.create_table('invoices', ..., sa.Column('id', sa.CHAR(length=32), ...))` will differ from what the app expects at runtime via asyncpg. Fix: use a temporary Postgres container (the plan offers this as an alternative) or accept that the migration file will need a one-line review after generation to fix UUID columns to `postgresql.UUID`.

- **LOW: `asyncpg==0.31.0` may conflict with `aiosqlite` in tests.** Both should install fine, but the plan should note that `aiosqlite` must also be in `requirements.txt` (it's mentioned in the action as a test lib without version pin). Since `requirements.txt` is the container requirements file, `aiosqlite` should be in a separate `requirements-dev.txt` or pyproject test group to avoid bloating the production image.

**Suggestions**

- Add `app/db/__init__.py` and a stub `app/db/models.py` to Task 1's file list with just `Base` and `SenderAllowlist` defined, then Task 2 fills in all remaining columns. This unblocks Task 1's verify step.
- Replace `sqlalchemy.dialects.postgresql.UUID` with `sqlalchemy.Uuid` throughout `models.py` to make tests work against SQLite without special handling.
- Add `requirements-dev.txt` with `pytest, pytest-asyncio, aiosqlite, httpx` to keep the production image lean.
- The acceptance criterion `grep "= None" backend/app/models/extraction.py | grep -vc '^#'` is brittle — it counts comment lines incorrectly (it negates lines starting with `#`, not inline comments). Use `grep -c "= None$"` instead.

**Risk Assessment: MEDIUM**

The plan will fail at Task 1's verify step due to the conftest import ordering issue. This is a blocking bug but straightforward to fix. The UUID/SQLite compatibility issue will surface in `test_db.py` (Task 2) when CRUD tests try to insert UUID primary keys. Both are fixable without architectural changes.

### Plan 02: Walking Skeleton (Docker + FastAPI + Vite + /health)

**Summary**

A solid orchestration plan that correctly implements the RESEARCH.md Docker patterns — healthcheck condition, anonymous node_modules volume, migrations-before-uvicorn entrypoint. The human checkpoint (Task 3) is the right gate for end-to-end verification. The main risks are around the Docker verify command's timeout assumptions and a subtle env-var-at-import-time issue in the health test.

**Strengths**

- All five Docker pitfalls from RESEARCH.md are explicitly addressed and mapped to acceptance criteria (can be audited by grep)
- The `condition: service_healthy` requirement for postgres is enforced via an acceptance criterion grep, not just documented
- Task 3 human checkpoint tests the INF-03 negative case (missing env var) end-to-end in a real container — this is the correct place for it
- Compose `environment: DATABASE_URL=...@postgres:5432/...` correctly overrides the `.env` value to use the Docker service hostname instead of `localhost`
- `autonomous: false` signals this plan requires the human verify gate — correct

**Concerns**

- **HIGH: `test_health.py` inherits the `engine.py` module-level import problem from Plan 01.** The instruction says "lazily import `app` inside the test function body" but the conftest.py `env_setup` fixture must have run first. pytest fixture ordering is by scope, not by test function body position. If `env_setup` is function-scoped and the app module has already been imported (e.g., because another test collected first), `Settings()` will have already been called without the right env vars. The cleanest fix is `autouse=True` on `env_setup` with `session` scope, combined with `get_settings.cache_clear()` after patching.

- **MEDIUM: `docker compose up -d --wait` default timeout is 30 seconds.** On the first run, both images need to build (not just start), Alembic runs migrations, and `pnpm install` inside the frontend container installs all node modules. The `--wait` flag times out per the Docker Compose default. If image builds exceed the wait window, the acceptance criterion will spuriously fail. Fix: either separate the `build` step (`docker compose build`) from the `up --wait` step (already done in the verify command as `build --quiet && ...up -d --wait`) or pass `--wait-timeout 120`.

- **MEDIUM: Vite proxy config `http://backend:8000` only resolves inside the Docker network.** If a developer runs `pnpm dev` locally (outside Docker) while the backend runs in a container accessible via `localhost:8000`, the proxy will fail DNS lookup on `backend`. Fix: use `http://localhost:8000` in the local vite.config.ts and rely on Docker's internal hostname aliasing only when inside the network. Alternatively, use an env var: `server.proxy: { '/health': process.env.VITE_API_URL || 'http://localhost:8000' }`.

- **LOW: `.env` ships with `sk-placeholder-replace-me` as `OPENAI_API_KEY`.** The value passes Settings validation (non-empty string), which is the intent for Phase 1. But if a developer forgets to replace it before Phase 2, the ExtractionService will fail with an OpenAI auth error rather than a clear startup message.

- **LOW: `docker compose exec -T postgres psql -U $POSTGRES_USER ...` in acceptance criteria.** The `$POSTGRES_USER` env var is in `.env`, not exported to the shell running the verify command. This will expand to an empty string. Fix: hardcode `compras` or use `$(grep POSTGRES_USER .env | cut -d= -f2)`.

- **LOW: `frontend/.dockerignore` isn't mentioned in the verify commands.** If it's missing, `COPY . .` in the frontend Dockerfile will include `node_modules/` from the host (if it exists), potentially overwriting the container-installed packages before the anonymous volume kicks in.

**Suggestions**

- Add `--wait-timeout 120` to `docker compose up -d --wait` in the verify command.
- Change the Vite proxy target to `http://localhost:8000` (or use an env var) so local non-Docker frontend dev works in Phase 4.
- Add `session`-scoped `autouse=True` env fixture to conftest to ensure Settings is initialized exactly once with test env vars before any test collection begins.
- Validate `.env` is gitignored in the Task 1 acceptance criteria via `git check-ignore -q .env && echo ok`.
- Add `backend/.dockerignore` to exclude `__pycache__/`, `.pytest_cache/`, and the generated `_autogen.db`.

**Risk Assessment: MEDIUM**

The plan correctly implements the target architecture. The main risks are operational (Docker timeout, proxy hostname) rather than architectural. The health endpoint test has a latent env-var import ordering issue that needs explicit fixture scoping.
