# Phase 2: Extraction Pipeline - Context

**Gathered:** 2026-05-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 2 delivers a fully tested `ExtractionService` that accepts an invoice image (as bytes + filename), calls GPT-4o vision via `client.chat.completions.parse()`, returns a structured `ExtractedInvoice` with a confidence score, stores the original image via `StorageBackend`, and persists the result to the database — with no WhatsApp dependency.

Deliverables:
- `backend/app/services/extraction.py` — `ExtractionService` class
- `backend/app/services/storage.py` — `LocalStorageBackend` implementing the `StorageBackend` abstraction
- `POST /extraction/test` endpoint (debug-only, gated on `settings.debug`)
- `backend/scripts/calibrate_prompt.py` — iterative prompt calibration script using Claude Opus 4.7 for ground truth
- `backend/tests/fixtures/` — real invoice image(s) + ground truth JSON files
- pytest suite covering ExtractionService with mocked OpenAI client + one `@pytest.mark.integration` live test

**This phase does NOT include:** WhatsApp webhook handling, sender allowlist checks, duplicate detection, or any admin UI — those are Phases 3 and 4.

</domain>

<decisions>
## Implementation Decisions

### Confidence Score
- **D-01:** Critical fields for confidence scoring are the four document header fields: `tipo_comprobante`, `numero_documento`, `proveedor`, `fecha`. Confidence score = proportion of these four fields that are non-null (range 0.0–1.0).
- **D-02:** Cross-field consistency checks are skipped in Phase 2. Base score only. Can be added as a v2 enhancement.
- **D-03:** Confidence threshold is `settings.confidence_threshold` (default `0.85`, already in `Settings`). Score ≥ 0.85 → `status=auto_saved`. Score < 0.85 → `status=pending_review`. Invoice is always saved — never rejected based on confidence alone.

### ExtractionService Interface
- **D-04:** `ExtractionService.extract(image_bytes: bytes, filename: str) -> ExtractionResult`. Returns `ExtractedInvoice` + `confidence_score` + `status` + `image_path`. Phase 3 passes WhatsApp media bytes directly; tests pass fixture file bytes.
- **D-05:** `POST /extraction/test` endpoint is registered only when `settings.debug is True`. In `create_app()`, the router is conditionally included. Local `.env` sets `DEBUG=true`; production leaves it `False`. No new env var needed.
- **D-06:** `POST /extraction/test` returns the full extraction result: `ExtractedInvoice` JSON + `confidence_score` + `status`. Does not persist to DB — developer testing surface only.

### StorageBackend
- **D-07:** Interface: `StorageBackend.save(data: bytes, filename: str) -> str`. Returns the stored relative path (written to `invoices.image_path`). Single method for Phase 2. `delete()` is Phase 4 scope.
- **D-08:** Storage root is configured via `STORAGE_PATH` env var added to `Settings` (default: `/data/invoices`). Docker Compose mounts a volume at this path.
- **D-09:** File naming convention: `{invoice_uuid}/{original_filename}`. Each invoice gets its own subdirectory named by UUID; original filename is preserved. Avoids collisions and keeps files organized per invoice.

### GPT-4o Prompt Strategy
- **D-10:** Initial prompt is minimal — start simple, let GPT-4o infer field mapping from the Pydantic schema. Calibrate iteratively using the validation dataset.
- **D-11:** Prompt calibration is a Phase 2 deliverable, not a post-phase manual step. `backend/scripts/calibrate_prompt.py` runs the full eval loop. Phase 2 is not complete until calibration passes on all fixture images.

### Prompt Calibration Workflow
- **D-12:** Ground truth is generated by **Claude Opus 4.7** (`claude-opus-4-7` via Anthropic SDK), not GPT-4o. Using GPT-4o for ground truth would create circular bias. Claude Opus 4.7 extracts the raw text content of each invoice image → saved as `tests/fixtures/{name}_ground_truth.json`.
- **D-13:** Calibration loop: (1) Run Claude Opus 4.7 on each fixture → save ground truth JSON. (2) Run GPT-4o structured extraction with current prompt → `ExtractedInvoice` JSON. (3) Field-by-field diff against ground truth. (4) Review findings, adjust prompt. (5) Repeat until no differences across all fixtures.
- **D-14:** `anthropic` Python SDK is a dev/scripts dependency only (not part of the main `requirements.txt`). Add to `requirements-dev.txt` or a `scripts/` requirements file.

### Test Strategy
- **D-15:** Mocked tests cover all logic: confidence score computation, status assignment, `StorageBackend.save()` calls, DB writes, error handling (extraction failure, storage failure). Fast, deterministic, no API cost.
- **D-16:** One `@pytest.mark.integration` live test uses a real fixture image and calls GPT-4o. Skipped by default; runs only with `pytest -m integration`. Requires `OPENAI_API_KEY` in environment.
- **D-17:** Fixture images live in `backend/tests/fixtures/`. Real invoice image(s) committed as part of Phase 2 implementation. Ground truth JSON files alongside them.

### Claude's Discretion
- Internal structure of `ExtractionResult` return type — researcher/planner picks the appropriate Pydantic response model.
- Whether `LocalStorageBackend` is a Protocol or ABC — pick the conventional Python approach.
- Exact system prompt text — start minimal, calibration loop refines it.
- Error types raised by `ExtractionService` on failure — pick meaningful exceptions.

</decisions>

<specifics>
## Specific Ideas

- The calibration script is a first-class Phase 2 deliverable — not an afterthought. Phase 2 is not done until `calibrate_prompt.py` runs clean against all fixture images.
- Claude Opus 4.7 (`claude-opus-4-7`) is the ground truth model. GPT-4o must not be used for ground truth generation (circular bias). The Anthropic SDK is a dev/scripts-only dependency.
- `/extraction/test` is a developer convenience, not a production feature. Gate it on `settings.debug` in `create_app()`.
- Confidence score is intentionally simple: `non_null_critical_fields / 4`. The four critical fields are `tipo_comprobante`, `numero_documento`, `proveedor`, `fecha`. Do not over-engineer in Phase 2.

</specifics>
